ES分词器及文本搜索分析

无标签
发布日期:   2023-02-05
文章字数:   2k
阅读次数:  

前言

工作中所负责的系统，包含丰富的搜索场景，其中就包含对标题及详情等文本类型字段的检索。用户经常会反馈搜索后的回收结果不准确，作为研发，就会对分词器、ES搜索关键字等进行调整。效果有时是有的，但很多时候是按下葫芦起了瓢。为了更好的优化搜索结果，有时会想去搞个自定义分词器；同时也对文本搜索的细节感到好奇。

周末正好有空学习下，本篇blog，预计会对以下问题进行分析：

  • 我们搜索操作时，对于搜索引擎，流程分几步？
  • ES分词器，输出结果是啥样的？如何计算输入的匹配度的？
  • ES有哪些常见分词器？如何自定义？
  • match/match_phrase/match_phrase_prefix/query_string区别？


ES文本解析的执行流程

首先明确何时进行文本解析？有两个时机：

  • 文档往ES索引的时候

  • 搜索文档的时候

ES官方认为大多数场景，应配套使用同一种解析器，避免分词或者常规化处理时的不匹配。

直接参考ES官网，文本解析宏观上就分为两步，Tokenization和Normalization。

Tokenization

Analysis makes full-text search possible through tokenization: breaking a text down into smaller chunks, called tokens. In most cases, these tokens are individual words.

很容易理解，就像程序编译需要先词法分析一样，分词打散再进行特征化处理是标准操作。

那么，分词后的输出是啥？我以最简单的分词器举例，对The QUICK brown-fox，分词结果如下：

{
  "tokens": [
    {
      "token": "The",
      "start_offset": 0,
      "end_offset": 3,
      "type": "word",
      "position": 0
    },
    {
      "token": "QUICK",
      "start_offset": 4,
      "end_offset": 9,
      "type": "word",
      "position": 1
    },
    {
      "token": "brown",
      "start_offset": 10,
      "end_offset": 15,
      "type": "word",
      "position": 2
    },
    {
      "token": "fox",
      "start_offset": 16,
      "end_offset": 19,
      "type": "word",
      "position": 3
    }
  ]
}

Normalization

这个常规化处理，就是解决大小写、单复数、同义词等同义场景问题的：

  • A search for Quick would not match quick, even though you likely want either term to match the other
  • Although fox and foxes share the same root word, a search for foxes would not match fox or vice versa.
  • A search for jumps would not match leaps. While they don’t share a root word, they are synonyms and have a similar meaning.

解决方法也简单粗暴：

To solve these problems, text analysis can normalize these tokens into a standard format. This allows you to match tokens that are not exactly the same as the search terms, but similar enough to still be relevant. For example:

  • Quick can be lowercased: quick.
  • foxes can be stemmed, or reduced to its root word: fox.
  • jump and leap are synonyms and can be indexed as a single word: jump.

上一步分词，已经排好了position还有起止偏移量，那么同义词怎么处理呢？

常见ES分词器

char_group

两个参数比较重要：

  • tokenize_on_chars：简单来说，就是分隔符
  • max_token_length：最大长度，默认255，超过就会被split

使用示例如下：

POST _analyze
{
  "tokenizer": {
    "type": "char_group",
    "tokenize_on_chars": [
      "whitespace",
      "-",
      "\n"
    ]
  },
  "text": "The QUICK brown-fox"
}

standard

默认分词器，按词切分。对于大部分语言都适用（中文不行，会单字切分）

同样有max_token_length参数，这里不赘述了。例子直接看[ES官网]([Standard tokenizer | Elasticsearch Guide 8.6] | Elastic)。

ik_smart/ik_max_word

这俩都是ik-analyzer这个解析器带的，是中文语境下最常用的分词器了。

ik_smart会做最粗粒度的拆分；已被分出的词语将不会再次被其它词语占有，而与之相反的，ik_max_word会将文本做最细粒度的拆分；尽可能多的拆分出词语。

挺抽象的，什么叫不能被其他词语占有呢？我拿联想是全球最大的笔记本厂商 举例：

  • ik_smart: [“联想”, “是”, “全球”, “最大”, “的”, “笔记本”, “厂商”]
  • ik_max_word: [“联想”, “是”, “全球”, “最大”, “的”, “笔记本”, “笔记”, “本厂”, “厂商”]

例如笔记本和本厂，由于本 已经被笔记本占有，所以不会再拆除本厂了。

path_hierarchy

这个在路径匹配场景非常好用，使用也非常简单：

// 请求
POST _analyze
{
  "tokenizer": "path_hierarchy",
  "text": "/one/two/three"
}

// 响应
[ /one, /one/two, /one/two/three ]

主要参数有已下4个：

  • delimiter：路径分隔符，默认是/。
  • replacement：分隔符的替换字符。
  • reverse：如果为true，则会倒置。
  • skip：忽略打头的n个路径节点。

举个栗子：

PUT my_index
{
  "settings": {
    "analysis": {
      "analyzer": {
        "my_analyzer": {
          "tokenizer": "my_tokenizer"
        }
      },
      "tokenizer": {
        "my_tokenizer": {
          "type": "path_hierarchy",
          "delimiter": "-",
          "replacement": "/",
          "skip": 2
        }
      }
    }
  }
}

// 请求
POST my_index/_analyze
{
  "analyzer": "my_analyzer",
  "text": "one-two-three-four-five"
}

// 响应
[ /three, /three/four, /three/four/five ]

如果reverse设置为true，则响应为：

[ one/two/three/, two/three/, three/ ]



如何自定义分词器

先说一下，分词器这个词有些狭隘，应该是解析器，它包含了分词器组件，且允许0到多个。解析器分为三大组件：

An analyzer — whether built-in or custom — is just a package which contains three lower-level building blocks: character filters, tokenizers, and token filters.

character filters，负责字符级别的替换；token filters，则是进行同义场景替换，就是normalization过程中提到的那些。分词器则是我们最熟悉的，会把一句话按照规则进行切割。

组件化带来的好处是，我们可以随意替换任何组件，或可以选取已有组件构成自己的解析器。

选择已有组件组装analyzer

PUT my-index-000001
{
  "settings": {
    "analysis": {
      "analyzer": {
        "my_custom_analyzer": {
          "type": "custom", 
          "tokenizer": "standard",
          "char_filter": [
            "html_strip"
          ],
          "filter": [
            "lowercase",
            "asciifolding"
          ]
        }
      }
    }
  }
}

POST my-index-000001/_analyze
{
  "analyzer": "my_custom_analyzer",
  "text": "Is this <b>déjà vu</b>?"
}

ES文本搜索的匹配逻辑

ES的fetch and query流程很经典，值得单独起一篇去讲，这里完整叙述所占篇幅可能太大了。本篇blog只分析文本搜索涉及的相关性计算等逻辑。query请求下发到了data节点后，lucene引擎就需要根据搜索关键字进行相关性排序。

一般来说，文本类搜索可能用到的关键字有match、match_phrase、match_phrase_prefix、query_string。这里我尽可能简单去描述执行流程。

match

match进行文本搜索，是非常模糊的。默认参数下，它不关注分词出现个数，也不关注出现顺序。大体上执行流程如下：

  • step-1: 将搜索输入进行分词。
  • step-2: 查询倒排链，找出各分词所指向的文档。
  • step-3: 计算匹配度，按score排序。

match_phrase

在使用默认参数的情况下，match_phrase的匹配非常严格。它要求包含所有分词，且顺序一致，且相邻(slop=0，大于1则允许不相邻)。执行过程如下：

  • step-1: 将搜索输入进行分词。
  • step-2: 查询倒排链，找出各分词所指向的文档。
  • step-3: 初筛，找到包含所有分词的文档。
  • step-4: 复筛，找到经过n次移动位置后，能与搜索输入匹配上的文档(n<=slot)

关于slot，什么是移动位置呢？可以参考下，图很简洁易懂：白话Elasticsearch18-深度探秘搜索技术之基于slop参数实现近似匹配

match_phrase_prefix

它的查询行为与 match_phrase 查询一致，但是它将查询字符串的最后一个词作为前缀(prefix)使用。

query_string

query_string搜索我认为非常灵活，它能帮助我们减少很多复合查询，比如要搜索name和content中，包含cavin或Lau，但不包含meituan的文档：

{
  "query": {
    "query_string": {
      "fields": ["name","content"],
      "query": "cavin OR lau -meituan"
    }
  }
}
  • +：表示必须包含的单词。
  • -：表示必须排除的单词。
  • AND：表示两个单词都必须出现。
  • OR：表示两个单词至少有一个出现。

不过要注意，假如我们query的是 cavin lau meituan ，那么会变为 cavin or lau or meituan .
